# 深度研究 + PPT生成 全链路 Skill

**TRIGGER KEYWORDS - Use this skill when user request contains ANY of these patterns:**
- (研究 OR 调研 OR 分析 OR research OR investigate OR study) AND (公司 OR 技术 OR 产品 OR company OR technology OR product)
- (深入 OR 深度 OR 全面 OR deep OR thorough) AND (研究 OR 调研 OR 分析 OR research)
- (研究 OR 调研) AND (报告 OR PPT OR 演示 OR report OR presentation)
- (生成 OR 做 OR 创建 OR 输出) AND (研究报告 OR 调研报告 OR PPT OR 演示文稿)
- "帮我研究" OR "研究一下" OR "调研一下" OR "深入了解"
- (了解 OR 分析) AND (这家公司 OR 这个技术 OR 这个产品 OR 这个方向)

**What this skill does:**
对任意课题（公司/技术/产品/行业方向）进行系统性深度研究，输出结构化Markdown研究报告（总览+分章节深度文档），最终生成带详细演讲备注的专业PPT。

**CRITICAL INSTRUCTION:**
When ANY trigger pattern matches, strictly follow the workflow below. 整个流程分为4个阶段，每个阶段都需要与用户交互确认后再进入下一阶段。

---

## Workflow Overview

```
阶段1: 信息采集 → 阶段2: 报告撰写 → 阶段3: 用户锤炼 → 阶段4: PPT生成
  ↓                  ↓                  ↓                  ↓
 抓取/搜索/整理      总览+分章节文档      多轮交互完善        python-pptx生成
```

---

## 阶段1: 系统性信息采集

### Step 1.1: 需求确认与研究计划

与用户确认研究目标，制定研究计划：
- 研究对象是什么？（公司/技术/产品/行业）
- 用户提供了哪些链接/线索？
- 用户是否提供了原材料（PDF/文档/笔记）？**原材料仅供参考，必须通过搜索补充和验证**
- 研究的深度和广度要求？
- 最终输出形式？（报告/PPT/两者都要）
- 目标受众是谁？（团队内部/客户/领导汇报）

创建TODO List跟踪研究进度。

### Step 1.2: 制定搜索研究计划（关键！先规划再执行）

**第一步：列出研究维度矩阵**

根据领域类型，从以下维度中选择适用的：

```
通用研究维度：
├── 基础概念：核心术语、分类体系、关键指标
├── 行业格局：头部玩家、市场规模、竞争态势
├── 技术栈：核心技术、工具链、架构模式
├── 方法论：业界最佳实践、标准流程、质量门
├── 痛点与挑战：行业公认的难题、未解决的问题
├── 前沿趋势：最新技术、新兴方向、颠覆性变化
├── 失败案例：常见陷阱、典型失败模式、教训
└── 成功路径：从入门到精通的路线图、关键里程碑

公司研究维度：
├── 基本面：概况/成立时间/团队/规模/地点
├── 产品线：所有产品/功能/性能指标/架构
├── 市场面：客户案例/市场份额/覆盖率/业务数据
├── 资本面：融资轮次/金额/投资方/估值
├── 竞争面：竞品对比/差异化优势/行业地位
├── 荣誉面：资质认证/行业榜单/奖项
├── 战略面：合作伙伴/发展历程/未来方向
├── 技术面：架构设计/性能基准/技术演进
└── 人才面：核心团队/招聘方向/技术栈偏好
```

**第二步：为每个维度构造搜索查询**

搜索查询构造公式：`{领域核心词} {维度关键词} {时间限定} {语言}`

**按信息类型选择搜索源：**

| 信息类型 | 推荐搜索源 |
|---------|-----------|
| 融资/估值 | CB Insights, PitchBook, Crunchbase |
| 行业排名 | Forbes, IDC, KPMG, Deloitte, Gartner |
| 学术前沿 | arxiv.org, SSRN, Google Scholar |
| 实践者经验 | Substack, Medium, 个人技术博客 |
| 开源项目 | GitHub, 官方文档 |

### Step 1.3: 多轮搜索执行（严格按节奏）

**第1轮（4路并行）：广度覆盖**
```
web_search × 4 并行：
  Query 1: {领域} 核心概念 + 分类体系
  Query 2: {领域} 行业格局 + 头部公司
  Query 3: {领域} 核心技术 + 方法论
  Query 4: {领域} 特定子领域（根据用户需求）
```

**第2轮（4路并行）：深度补充**
```
web_search × 4 并行：
  Query 5: {领域} 前沿趋势 + AI/新技术应用
  Query 6: {领域} 痛点 + 挑战 + 失败案例
  Query 7: {领域} 工具框架 + 开源项目
  Query 8: {领域} 用户特定关注点
```

**递归深入原则：** 从搜索结果中发现新的重要话题时，立即追加搜索。
例如：搜到"Alpha Decay"概念 → 追加搜索"alpha decay quantitative trading strategy lifecycle"

**第3轮（按需，2-4路）：针对性补充**
- 针对第1-2轮中发现的关键话题深入搜索
- 针对用户特定背景补充搜索

### Step 1.4: 深度阅读（关键！不可跳过！）

从所有搜索结果中，筛选 **3-5篇最有深度的文章** 用 `web_fetch` 或 `firecrawl_scrape` 全文阅读。

**选择标准（优先级从高到低）：**
1. 实践者的第一手经验
2. 含具体数据点和技术细节的分析文章
3. 头部公司/权威机构的深度报告
4. 学术论文的核心发现
5. 避免：纯理论综述、营销软文、过于浅显的入门文章

**阅读时重点提取：**
- 具体数字（"年均衰减5.6%"、"10个月后转负"）
- 技术架构/模型
- 实践者的失败教训
- 流程/框架
- 对比数据

### Step 1.5: 官网/主要来源深度抓取

**第一步：主页 + 站点地图（并行执行）**
- `firecrawl_scrape` 抓取主页（formats: markdown）
- `firecrawl_map` 发现所有子页面

**第二步：逐个抓取所有子页面（并行，每批5-8个）**
- 产品页：每个产品单独抓取
- 关于页：公司简介、发展历程、资质荣誉
- 新闻页、招聘页、技术博客/社区等

**第三步：覆盖率检查（关键！不可跳过）**

对照 `firecrawl_map` 返回的URL列表，逐一确认是否已抓取。
发现遗漏立即补充。这一步决定了报告的完整性。

**第四步：记录所有图片URL**
从抓取内容中提取所有图片链接，按主题分类保存，供后续文档引用。

### Step 1.6: 用户原材料处理

如果用户提供了原材料（PDF/文档/链接/笔记）：
1. **全部阅读**，提取关键信息
2. **不盲信**，用搜索结果交叉验证
3. **发现缺口**，原材料中提到但未展开的话题，主动搜索补充
4. **发现矛盾**，原材料与搜索结果不一致时，标注并向用户确认

### Step 1.7: 信息汇总确认

- 向用户汇报采集结果和关键发现
- **先在对话中输出完整研究报告草稿**（让用户看到全貌）
- 等待用户确认后再进入文档落地阶段

## 阶段2: 结构化报告撰写

### Step 2.1: 创建项目目录

在用户指定目录（默认 `~/Code/{课题}-research/`）下创建：

```
{课题}-research/
├── README.md                    # 索引导航+核心数据速览
├── 00-{课题}总览.md              # 主报告
├── 01-{子主题1}.md              # 深度章节
├── 02-{子主题2}.md              # 深度章节
├── ...
└── ppt/                         # PPT生成脚本
    ├── h.py                     # Helper函数
    └── build_pN.py              # 分段构建脚本
```

### Step 2.2: 撰写总览文档 (00-总览.md)

必须包含：
- 基本信息表格
- 融资/资本情况
- 产品/技术体系总览表格（名称/定位/核心指标）
- 市场地位关键数据
- 资质荣誉列表
- 发展历程时间线
- 战略合作伙伴
- 深度章节索引表

### Step 2.3: 撰写深度章节 (01~0N)

**每个章节必须包含：**
1. 产品/技术定位（一句话+详细描述）
2. 核心性能指标表格
3. 技术架构解析（ASCII架构图 + 引用官网图片）
4. 功能特性详解（含图片引用）
5. 应用场景
6. 业务成果数据
7. 竞品对比（如适用）
8. "给团队的关键认知"总结（3-5条要点）

**图片引用：** 所有官网图片用 `![描述](URL)` 格式嵌入。

**文件大小控制：** 每个文件200-350行。用 `cat > file << 'EOF'` heredoc写入避免超时。

### Step 2.4: 完整性检查

```bash
# 检查产品覆盖
for p in "产品1" "产品2" ...; do grep -rl "$p" *.md | wc -l; done
# 检查图片引用数
grep -c "官网域名" *.md
# 统计总量
wc -l *.md && du -sh .
```

向用户展示检查结果。

## 阶段4: PPT生成

### Step 4.1: 确定设计风格

**核心原则：PPT的视觉风格必须匹配研究主题的调性。**

根据研究主题自动选择配色：

| 主题类型 | 底色 | 主色 | 强调色 | 视觉调性 |
|---|---|---|---|---|
| 金融科技 | #0A1A2E深蓝 | #00D4FF青 | #FFB800金 | 科技感+信任感，深色底+霓虹高亮 |
| AI/ML | #0F0A2E深紫 | #A85CFF紫 | #00E676绿 | 未来感+智能感，渐变紫+荧光绿 |
| 云计算 | #232F3E | #FF9900橙 | #5ACBF0蓝 | 专业稳重，AWS风格 |
| 区块链 | #0A0A1A | #00FF88绿 | #FF4444红 | 赛博朋克，暗黑+霓虹 |
| 通用商业 | #1A2332 | #4A90D9蓝 | #F5A623橙 | 商务专业 |

**炫酷设计要素（必须应用）：**

1. **深色背景 + 高对比度高亮色** — 所有幻灯片使用深色底，关键数字用亮色（青/金/绿）高亮
2. **大数字指标（metric）** — 核心数据用32-36pt粗体+主题色，视觉冲击力强
3. **圆角卡片布局** — 内容分区用半透明深色卡片，层次分明
4. **顶部装饰线** — 每页标题下方用主题色细线分隔，统一视觉节奏
5. **章节分隔页** — 深色背景+"第N章"小标题+大标题，营造节奏感
6. **颜色编码** — 不同产品/维度用不同颜色区分（青/金/绿/紫），信息层次清晰
7. **封面页** — 大标题居中+装饰线+副标题，简洁有力
8. **数据仪表盘** — 多个metric卡片网格排列，一眼看到全貌

### Step 4.2: 创建Helper函数 (ppt/h.py)

标准函数库，包含：
- `bg(slide, color)` — 设置背景色
- `t(slide, left, top, w, h, text, size, color, bold, align)` — 添加文本框
- `mt(slide, left, top, w, h, lines)` — 多行文本（lines为元组列表）
- `bar(slide, left, top, w, h, color)` — 装饰线条
- `card(slide, left, top, w, h, color)` — 圆角卡片
- `hdr(slide, title, subtitle)` — 页面标题栏+分隔线
- `ftr(slide)` — 页脚（主题+日期）
- `n(slide, text)` — 设置演讲备注
- `metric(slide, left, top, number, label, color)` — 大数字指标
- `chap(prs, num, title, subtitle)` — 章节分隔页

配色常量根据Step 4.1的选择设置。字体统一用Calibri。幻灯片尺寸16:9（16×9英寸）。

### Step 4.3: 分段构建PPT

**关键防超时策略：**
- 每个build脚本只生成5-8页幻灯片
- 每段脚本读取上一段的.pptx文件，追加新幻灯片后保存
- 每段运行后立即验证页数

**构建链：**
```
build_p1.py → archforce_p1.pptx (封面+概况, ~6页)
build_p2.py → archforce_p2.pptx (核心技术, ~7页)
build_p3.py → archforce_p3.pptx (更多产品, ~6页)
build_p4.py → 最终输出.pptx    (总结, ~6页)
```

**每个build脚本的结构：**
```python
from pptx import Presentation
# 内联定义所有helper函数（避免import路径问题）
# 读取上一段的pptx
prs = Presentation('上一段.pptx')
# 添加幻灯片...
# 保存
prs.save('本段.pptx')
print(f"Done: {len(prs.slides)} slides")
```

### Step 4.4: 标准PPT结构

| 页码 | 类型 | 内容 |
|---|---|---|
| 1 | 封面 | 标题+副标题+日期 |
| 2 | 目录 | 所有章节（编号+中英文） |
| 3 | 章节分隔页 | "第N章"+标题 |
| 4-N | 内容页 | 指标仪表盘/产品矩阵/对比表/架构图 |
| 末页 | 总结 | 核心结论+关注点+展望 |

**常用页面布局模板：**

1. **指标仪表盘页：** 顶部6个metric大数字 + 下方详细说明卡片
2. **产品矩阵页：** 横排3-5个card，每个含名称/定位/指标/特色
3. **对比表格页：** 表头行+数据行，自家产品列用绿色高亮
4. **架构图页：** 上方标题+中间大面积架构图/流程图+下方说明
5. **章节分隔页：** "第N章"+大标题+副标题，深色背景

### Step 4.5: 演讲备注规范（极其重要！决定PPT质量的关键）

**字数硬性要求：**
- 内容页（数据/产品/对比/架构页）：**200-500字**
- 封面页/开场白：**200-300字**（含自我介绍、演讲目标、全局概述）
- 目录页：**200-300字**（逐条解释每章要讲什么、为什么这样安排）
- 章节分隔页：**50-100字**（过渡语+下一章预告）
- 总结页：**300-500字**（核心结论展开+行动建议+结束语）
- **绝对下限：任何页面不得少于50字**

**每页备注必须包含的5个要素：**
1. **【标题标记】** 用"【XXX】"开头标记页面主题
2. **核心讲解** — 不是复述幻灯片文字，而是展开解释背后的含义和逻辑
3. **数据解读** — 每个关键数字都要解释"这意味着什么"、"为什么重要"
4. **类比/对比** — 用通俗的类比帮助理解（如"Kafka是高速公路，AMI是光速通道"）
5. **衔接过渡** — 与上一页的承接、向下一页的引导

**备注写作风格：**
- 口语化，像在跟同事面对面讲解
- 用引导词："大家注意"、"这里的关键是"、"简单说"、"换个角度看"
- 数据要解读含义，不只是念数字（"95%覆盖率意味着几乎是行业事实标准"）
- 技术概念要用类比解释（"共享内存60ns，相当于CPU访问L3缓存的速度"）
- 竞品对比要点明差异本质（"不是优化问题，而是架构本质不同"）
- 章节分隔页写过渡语（"刚才讲了XX，现在进入YY，这两者的关系是..."）

**备注质量硬性目标：**
- 内容页平均字数 **≥ 200字**
- 总字数 **≥ 5000字**（25页PPT至少5000字备注）
- 每页都有备注，**无一遗漏**
- 拿着备注就能完成一场**30-45分钟**的完整演讲
- 验证时如果发现某页备注<100字，必须补充到200字以上

**优秀备注示例（供参考）：**

```
【AMI核心性能】
AMI的性能指标非常惊人：

1. 共享内存传输时延60纳秒——这是什么概念？CPU访问L3缓存大约40纳秒，
AMI几乎实现了零开销传输。这是通过无锁环形缓冲区、CPU亲和性绑定、
内存预分配等技术实现的。

2. 跨主机传输时延950纳秒——10G以太网单跳约500纳秒，说明AMI的软件栈
开销极小。支持RDMA直接绕过操作系统内核。

3. 吞吐量3500万消息/秒——超高并发能力。

三种通信模式是AMI的核心创新：
- 共享内存：同机进程间，60ns
- 可靠组播：一对多分发，行情场景关键
- TCP/RDMA：跨机通信，支持RDMA绕过内核

简单说，Kafka是"高速公路"，AMI是"光速通道"。它们解决的是完全不同
层次的问题。AMI比Kafka快4-5个数量级，这不是优化问题，而是架构本质不同。
```

## 质量标准

### 研究报告
- 所有产品/技术/数据点全覆盖

## 防超时策略

1. Markdown文件：用 `cat > file << 'EOF'` heredoc写入
2. Python脚本：每个脚本内联helper函数，不依赖import
3. PPT分段构建：每段5-8页
4. 如果工具报"too large"，立即拆分为更小步骤

## Notes

- 整个流程可能需要30-60分钟
- 阶段1-2可连续执行，阶段3需要用户参与
- PPT生成前必须确认报告内容已完善
- 所有中间产物（Markdown、Python脚本）都保留方便迭代
- 如果用户只要报告不要PPT，可在阶段2结束后停止
